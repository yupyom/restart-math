## 1. この設計で解決したいこと

この教材の中心的な課題は、数学に苦手意識のある学習者が、画面に多くの説明・新しい文字・複数の選択肢を同時に見てしまい、何から読めばよいか分からなくなることです。

そこで、教材を単に「単元の一覧」にせず、次の四つの役割に分けます。

| 役割 | 学習者がすること | 画面 | 例 |
| --- | --- | --- | --- |
| 範囲 | 今いる場所を知る | 範囲 | 「中学の平方根の復習。数 I の実数へつながる」 |
| 単元 | 一つの考え方を短く理解する | 単元ノート | 「同類項は同じ種類のカードだけをまとめる」 |
| 図解 | 物や動きで規則を確かめる | 動く図解 | x カード、x 袋、面積、数直線 |
| 問題 | 一手ずつ自分で使う | ステップ問題 | 2 桁以内を中心にした分配法則の問題 |

一つの画面でこの四つを同時に見せず、**単元を中心にして、必要な図解と問題へ一手で移動できる**ようにします。

## 2. 学習の基本導線

各単元は、必ず同じ順に進めます。

```text
範囲を知る
    ↓
短い単元ノート（何を覚えるか）
    ↓
具体物の図解（なぜそう見えるか）
    ↓
初歩のステップ問題（自分で一手使う）
    ↓
次の単元 / 復習マップ
```

例: 一次方程式

```text
中1「方程式」
    ↓
「左右から同じ 1 ブロックを取る」
    ↓
x 袋と 1 ブロックの図解
    ↓
3x - 2 = 10 を、まず +2、次に ÷3 の二手に分ける問題
```

この順番では、最初に文字だけの式を見せません。`x` は、図解では x 袋または x カードとして先に登場し、その後で式の表記に対応させます。

## 3. コンテンツの正本と公開物

編集しやすさと GitHub Pages の単純さを両立するため、次の二層に分けます。

```text
リポジトリ
├── src/                         # 編集する正本
│   ├── index.html                # 画面の骨組み
│   ├── assets/
│   │   ├── css/                  # 見た目
│   │   └── js/                   # 画面描画・図解・問題の動き
│   └── content/                  # 教材データ（構造化された正本）
│       ├── lessons.js            # 単元ノートの構造・関連づけ・表示ブロック
│       ├── labs.js               # 図解の説明・関連単元
│       ├── practice.js           # 問題モードの設定
│       ├── stories.js            # 理由・社会との接続を読む補助教材
│       └── topics.js             # 学習マップのカード
├── scripts/
│   ├── build.mjs                 # src/ を docs/ へコピーして検証
│   └── validate-content.mjs      # ID・リンク・数式表記の検査
├── design/                       # このような設計文書（公開物ではない）
├── docs/                         # GitHub Pages が配信する完成物
└── README.md                     # 最小限の導入・操作・公開手順
```

`docs/` は手で直接編集しません。`src/` の変更後にビルドして更新する公開用の成果物です。これにより、公開済みの HTML を直接直す際の修正漏れを防ぎます。外部のビルド用依存関係は増やさず、Node.js の標準機能だけで生成できるようにします。

現時点の短い単元ノートは、リンク ID・図解設定・数式・表示ブロックを一緒に検査できる構造化 JavaScript を正本にします。これはブラウザが読み込む小さなデータモジュールであり、編集時・実行時ともに負荷の問題にはなりません。むしろ Markdown へ一律に移すと、段階式・文章題・図の説明を再び文字列として曖昧に扱う危険があります。

一方、300字を超える読み物や出典付きの長文は Markdown を採用できるようにします。ただし移行の条件は「Markdown にした」ことではなく、見出し・段落・数式・出典・図の参照をビルド時に検査できることです。短い単元の構造化データと、長文の Markdown を使い分けるハイブリッドを基本方針にします。

## 4. データの分け方

### 4.1 単元 (`lessons.js`)

単元は教材本文の最小単位です。各単元には、必ず範囲・学習段階・短い説明・関連図解・関連問題を持たせます。

```js
{
  id: "linear-equations",
  order: 13,
  range: ["中1"],
  strand: "数と式",
  stage: "方程式 1",
  title: "一次方程式を解く",
  summary: "左右から同じブロックを取り…",
  points: ["…", "…"],
  example: {
    type: "aligned-steps",
    rows: ["3x-5=16", "3x=21", "x=7"]
  },
  objectIntro: "x袋と1ブロックで考える",
  labIds: ["equation-blocks"],
  practiceIds: ["linear-equation"],
  nextLessonId: "simultaneous-equations"
}
```

`range` は学校段階（`中1`、`中3`、`数I`、`数A`）だけに使い、`stage` は単元内の順番だけに使います。たとえば「数I」が二つ並んで別の意味に見える、といったラベルの混乱を避けます。

### 4.1.1 本文と例の表示ブロック

単元本文を単なる HTML 断片や「数式を含む一つの文字列」にはしません。見た目を決める前に、教材上の役割を次の型で明示します。

| 型 | 使う場面 | 表示の約束 |
| --- | --- | --- |
| ふつうの説明 | 定義・要点・図の読み方 | 文中の変数だけをインライン数式にする |
| `aligned-steps` | 計算過程・互除法 | 一行ごとに等号をそろえる。別の式を一つの連鎖にしない |
| `word-problem` | 文章題から式を作る例 | 問題文、量の意味、最後の式を分けて示す |
| `narrative` | 図形の説明・短い比較 | 文章を先に読み、必要な結論式だけを独立表示する |

たとえば複数の等式をカンマでつないだ文章を自動で縦計算に変換してはいけません。互除法なら各行が別の等式、分配法則の途中式なら同じ式の変形、文章題なら式の前に状況説明が必要です。この区別をデータで持ち、表示側の推測に任せないことを品質基準にします。

### 4.2 図解 (`labs.js`)

図解は「遊べる部品」ではなく、特定の誤解をほどくための教材です。`id`、対応単元、初めに見せる具体物、確かめる不変量をデータとして持ちます。

```js
{
  id: "equation-blocks",
  title: "x袋と1ブロック",
  category: "方程式",
  lessonIds: ["linear-equations"],
  objectIntro: "同じ数の1ブロックを、左右から同時に取る",
  observe: "左右の合計はいつも同じ",
  hostId: "equation-lab"
}
```

図解の表示と実際の操作コードは分けます。上のデータは「どの単元から何の図解へつなぐか」を担当し、キャンバス描画やスライダー処理は、現時点では `assets/js/app.js` にあります。今後この処理を `assets/js/labs.js` などへ分けても、教材データの参照方法は変えません。

#### 4.2.1 応用図解の安全な扱い

ローン・会計・速さ・データのように日常や社会に接続する図解は、具体的な判断を勧めるページにはしません。画面で扱うのは、次のような一つの数値関係だけです。

| 題材 | 確かめる関係 | 画面に明記する範囲 |
| --- | --- | --- |
| ローン | 元金 × 年率 × 年数（単利の教材モデル） | 実際の契約・返済方式・商品比較には使わない |
| 会計 | 売上 − 変動費 − 固定費 = 利益 | 税金・在庫・事業判断は扱わない |
| 速さ | 道のり = 速さ × 時間 | 速さが一定の場面に限る |
| データ | 同じ平均でも散らばりは変わる | まず最小・最大の差を見て、分散・標準偏差は別単元へつなぐ |

各図解は、スライダーの現在値、図、説明文を同時に更新します。数だけを動かして「何を変えたのか」が分からなくならないことを、操作の必須条件にします。

### 4.3 問題 (`practice.js`)

問題は「出題器の設定」と「採点・解説の処理」を分けます。設定側には、初期難度・数の上限・前提単元・関連図解を置きます。

```js
{
  id: "simplify-radical",
  title: "平方数を外へ出す",
  lessonIds: ["simplify-roots"],
  labIds: ["radical-simplifier"],
  level: "はじめの一問",
  numberPolicy: "radicand-under-100",
  generator: "generateRadicalProblem"
}
```

最初の問題は、原則として答えや途中計算が 2 桁程度に収まる設定にします。問題を解けない理由が、規則の理解ではなく繰り上がり・九九・大きな数の分解にならないようにするためです。慣れてから選べる「少し進んだ問題」を別に追加します。

### 4.4 学習マップ (`topics.js`)

学習マップは新しい本文を持たず、単元への入口です。単元データの `id` を参照し、重複して説明を抱えません。

```js
{
  id: "topic-linear-equations",
  lessonId: "linear-equations",
  category: "数と式",
  level: 1,
  label: "一次方程式"
}
```

これにより、単元名や範囲の修正は一か所で済みます。

### 4.5 読み物 (`stories.js`)

読み物は、単元本文を長くせずに「なぜその約束が必要か」を補う教材です。`type`、対応する単元・図解・問題、本文、出典、事実確認の状態を持たせます。

```js
{
  id: "equal-sign-keeps-balance",
  type: "rule",
  lessonIds: ["identities-equations", "linear-equations"],
  labIds: ["equation-lab"],
  practiceIds: ["equation"],
  sources: [],
  factCheck: { status: "checked", note: "外部の事実を含まない" }
}
```

人物史・社会・統計を扱う `history` / `society` の読み物では、`sources` に HTTPS の出典を少なくとも一つ入れます。出典と `factCheck.status` はビルド前の内容検査で確認します。

## 5. 相互リンクのルール

リンクはすべて固定 ID を介して作り、表示文言や配列の順番へ依存させません。

| 出発点 | 行き先 | 表示する導線 |
| --- | --- | --- |
| 範囲 | 単元 | 「この範囲の単元を見る」 |
| 単元 | 図解 | 「x袋と1ブロックを動かす」 |
| 単元 | 問題 | 「この単元の一問を解く」 |
| 図解 | 単元 | 「説明を読む」 |
| 図解 | 問題 | 「この図解のルールで解く」 |
| 問題 | 図解 | 「図解に戻る」 |
| 学習マップ | 単元 | 「単元を開く」 |

一つの単元に関連する図解・問題が複数ある場合でも、最初に表示するのは「おすすめ 1 つ」です。残りは「ほかの図解」「練習を増やす」にしまい、スマートフォンで選択肢が増えすぎないようにします。

## 6. 画面構成とスマートフォンでの優先順位

ハッシュで開くページを、次の七つに整理します。

| URL | 役割 | 画面上で最初に見せる内容 |
| --- | --- | --- |
| `#home` | はじめに | 今日の入口を一つだけ |
| `#scope` | 範囲 | 今の単元が中学・数I・数Aのどこか |
| `#lessons/<lesson-id>` | 単元 | 見出し、3つ以内の要点、例、次の一手 |
| `#labs/<lab-id>` | 図解 | 具体物、操作、観察すること |
| `#practice/<practice-id>` | 問題 | 一問、今の手順、ヒント |
| `#stories/<story-id>` | 読み物 | ルールの役割、短い確認、教材へ戻る導線 |
| `#map` | 学習マップ | 次に選べる少数の単元 |

現行の `#lessons` のような大分類ハッシュも残し、旧リンクを壊しません。詳細ルートは、たとえば `#lessons/linear-equations` とします。

### モバイルの原則

1. 一画面に一つの主目的だけを置く。
2. 単元本文は「要点 3 個・例 1 個・次の一手 1 個」を基本とする。
3. 長い単元一覧は、常時表示せず「単元を選ぶ」ボタンから開く。
4. ページ送りは現在地と前後一つずつを優先し、全ページ番号は補助にする。
5. 図解は説明を読んでから操作できる順序にし、キャンバスだけを先に置かない。

## 7. 数式表記を壊さないための規約

数式は MathJax の TeX として保存します。表示用 HTML を組み立てるときに `<` や `>` が HTML タグとして扱われないよう、本文・例・要点・ラベルは必ず `escapeHtml()` を通します。

さらに、ビルド時に次を検査します。

- 単元・図解・問題・学習マップの ID が重複していないこと
- 参照される `lessonId`、`labId`、`practiceId` が必ず存在すること
- すべての単元に少なくとも一つの「次の一手」があること
- 数式を含む文字列に生の HTML タグが混ざっていないこと
- `docs/` の `index.html` がすべてのローカル CSS・JS を参照できること

## 8. 移行の順番

1. この設計文書を正本にし、出力先を `docs/` に決める。
2. 現在の静的ファイルを `src/` へ移し、同じ内容を `docs/` へ生成できるようにする。
3. `units`、`topics`、図解カタログ、問題モード設定を `src/content/` へ切り出す。
4. `app.js` を、ルーティング・表示・図解・問題生成の役割ごとに分ける。
5. 相互リンクを ID ベースへ統一し、旧ハッシュへの互換を残す。
6. ビルドと内容検査を通してから、GitHub Pages の対象を `main /docs` に変更する。

この移行中は `docs/` が公開物ですが、GitHub にはプッシュしません。公開前に、リンク切れ・数式・スマートフォンでの単元導線をまとめて確認します。

## 9. 完了の目安

- `src/` だけを編集すれば、`docs/` の完成物を再現できる。
- 単元・図解・問題のどこからでも、関連する次の教材へ移れる。
- 初心者向けの問題設定がデータとして明示されている。
- 数Iと数Aの範囲ラベルと、単元内の順序ラベルが混ざらない。
- GitHub Pages は `main` ブランチの `/docs` を配信するだけで動く。

## 10. 表記と文字組みのルール

画面上の役割で書体を分けます。目次・ボタン・範囲ラベル・見出しは、選びやすさを優先してゴシック系にします。説明文・要点・例・ヒントは、PCでは18pxを基準にした明朝系とし、落ち着いて読める行間を確保します。

変数・演算・関数名は、文中であってもMathJaxの数式書体にそろえます。たとえば `x袋` は「数式の `x` + 日本語の袋」として表示します。教材データにすでに含まれるTeX表記はそのまま保持し、プレーンテキストの孤立した `x`、`y`、`a`、`b`、`n`、`sin`、`cos`、`tan` は表示時に数式化します。

ただしHTMLの`select`の選択肢内はMathJaxが組版しないため、数式を厳密に見せる必要がある操作は、将来ボタン型の選択UIなどへ移します。

## 11. ルールの理由・背景・社会との接続

### 11.1 前提

この教材では、公式や定義を「すでにある決まり」としてだけ提示しません。学習者が「なぜそう決めるのか」で止まることを、理解不足ではなく自然な問いとして扱います。

各ルールは、可能な範囲で次の順に短く示します。

```text
困る場面・そろえたいこと
    ↓
共有する約束・定義
    ↓
その約束で解けること
    ↓
手を動かす小さな確認
```

たとえば計算順序は、「同じ式を見た人が同じ答えにたどり着けるようにする共通の読み方」として置きます。かっこは、その共通の読み方の一部を意図的に変更するときの印です。ここで扱うのは厳密な数学史の断定ではなく、学習者がルールの役割をつかむための説明です。歴史的事実を述べる場合は、出典を確認し、後づけの単純な物語にしません。

### 11.2 単元に追加する文脈データ

単元データには、本文とは別に次の短い要素を持たせます。すべてを毎回開かず、必要な子が「なぜ？」を開ける構成にします。

```js
{
  id: "integer-rules",
  // 既存の title, summary, points ...
  context: {
    why: {
      question: "どうして計算する順番を決めるの？",
      answer: "式を読む人によって答えが変わらないよう、共通の読み方を決めているから。",
      tryIt: "2+3×4 を、先に足す場合と先にかける場合で比べる"
    },
    definitions: [
      { term: "式", meaning: "数や記号を、計算の約束に従って並べたもの", boundary: "等号で『成り立つか』を問うと方程式になる" }
    ],
    connections: [
      { kind: "daily-life", title: "会計の合計", summary: "計算順序をそろえると、複数の人が同じ集計結果を確かめられる" }
    ],
    storyIds: ["shared-notation"]
  }
}
```

`why` は「ルールの役割」を、`definitions` は「言葉をどう使うか・どこまでを指すか」を、`connections` は「何を扱えるようになるか」を担当します。正解を出すために全てを読ませるのではなく、立ち止まったときの補助線にします。

### 11.3 画面の置き方

単元ページでは、要点の直後に常時長文を置きません。次の二つを短い開閉式のカードとして置きます。

| カード | 入口の文 | 中身 | 量 |
| --- | --- | --- | --- |
| なぜこの約束？ | 「もし決めなかったら、何が困る？」 | 困る場面 → 約束 → 小さな比較 | 2〜4文 + 1操作 |
| どこで役立つ？ | 「この考え方で、何が見える？」 | 身近な場面・科学・仕事への接続 | 2〜4文 + 図解リンク |

用語は別の小カード「言葉をほどく」にまとめ、定義だけで終わらせず、使う例と使わない例を一つずつ示します。読み物を開かない学習者でも単元と問題は進められ、読み物を開く学習者には理解の出口を増やす設計です。

### 11.4 読み物ページ

単元とは別に `#stories/<story-id>` の読み物ページを設けます。読み物は単元順の長い連続講義にせず、次の三種類を混ぜます。

| 種類 | ねらい | 例 |
| --- | --- | --- |
| ルールの物語 | 約束が必要になる場面を知る | 計算順序、負の数、等号、座標 |
| 人と発見の物語 | 数学者・科学者がどんな問いを扱ったか知る | デカルトと座標、ガウスと数、フローレンス・ナイチンゲールと統計 |
| 社会と仕事の物語 | 数学が現代の判断・説明にどう使われるか知る | ローンの返済、利益と損益分岐、データのばらつき、測定と誤差 |

人物史は「一人の天才が突然作った」とは見せず、当時の問題・先行する考え・今の使われ方を一緒に扱います。金融や医療などの題材は、個別の意思決定を勧めるものではなく、数値の関係を読む練習として扱います。

### 11.5 動かせる応用例

難しい現実の話題も、最初は数値を操作するだけで関係が見える図解にします。

- ローン: 元金・年率・回数のバーを動かし、月々の返済額と総返済額の違いを見る
- 会計: 売価・原価・固定費・販売数のバーを動かし、利益が0になる点を読む
- 科学: 距離・時間・速さを動かし、表・グラフ・式が同じ関係を示すことを見る
- データ: 外れ値を一つ動かし、平均・中央値・散らばりがどう変わるかを見る

各図解は、数式を先に覚えさせるためではなく、「何を変えると何が動くか」を観察するために置きます。数値は初めは小さく、単位・前提・現実との違いも短く明示します。

### 11.6 追加時の品質基準

1. 「なぜ？」への答えは、権威ではなく困る場面・共通の目的・使えるようになることから始める。
2. 定義には、使う例と境目の例を添える。
3. 歴史・人物・社会の事実は、信頼できる一次資料・公的機関・博物館・大学などで確認し、読み物に出典を示す。
4. 一枚に情報を詰め込まず、単元本文・理由カード・応用図解・読み物を分ける。
5. 読み物を読まないことが問題演習の障害にならないようにする。
