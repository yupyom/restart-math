## 1. この設計で解決したいこと

この教材の中心的な課題は、数学に苦手意識のある学習者が、画面に多くの説明・新しい文字・複数の選択肢を同時に見てしまい、何から読めばよいか分からなくなることです。

そこで、教材を単に「単元の一覧」にせず、次の四つの役割に分けます。

| 役割 | 学習者がすること | 画面 | 例 |
| --- | --- | --- | --- |
| 範囲 | 今いる場所を知る | 範囲 | 「中学の平方根の復習。数 I の実数へつながる」 |
| 単元 | 一つの考え方を短く理解する | 単元ノート | 「同類項は同じ種類のカードだけをまとめる」 |
| 図解 | 物や動きで規則を確かめる | 動く図解 | x カード、x 袋、面積、数直線 |
| 問題 | 一手ずつ自分で使う | ステップ問題 | 2 桁以内を中心にした分配法則の問題 |

一つの画面でこの四つを同時に見せず、**単元を中心にして、必要な図解と問題へ一手で移動できる**ようにします。

## 2. 学習の基本導線

各単元は、必ず同じ順に進めます。

```text
範囲を知る
    ↓
短い単元ノート（何を覚えるか）
    ↓
具体物の図解（なぜそう見えるか）
    ↓
初歩のステップ問題（自分で一手使う）
    ↓
次の単元 / 復習マップ
```

例: 一次方程式

```text
中1「方程式」
    ↓
「左右から同じ 1 ブロックを取る」
    ↓
x 袋と 1 ブロックの図解
    ↓
3x - 2 = 10 を、まず +2、次に ÷3 の二手に分ける問題
```

この順番では、最初に文字だけの式を見せません。`x` は、図解では x 袋または x カードとして先に登場し、その後で式の表記に対応させます。

## 3. コンテンツの正本と公開物

編集しやすさと GitHub Pages の単純さを両立するため、リポジトリを **「正本 `src/`」と「公開物 `docs/`」の二層**に分けます。触って編集するのは常に `src/` だけです。

### 3.1 生成モデル：`docs/` は `src/` の完全コピー（バンドルなし）

`docs/` は手で編集しません。`npm run build`（`scripts/build.mjs`）が `src/` から作り直します。生成は**検証付きの丸ごとコピー**で、結合・変換（バンドル）は一切行いません。

```text
npm run build ＝ scripts/build.mjs
  1. validateContent(src)   … 生成前に src を検査（ID重複・逆参照・数式など）
  2. rm -rf docs/           … 公開物をいったん全消去
  3. cp -r src/ → docs/     … src を丸ごとコピー（変換・結合なし）
  4. docs/.nojekyll を作成  … GitHub Pages 用
  5. validateContent(docs)  … 生成後の docs も同じ検査
```

ここから次の**不変条件**が導かれます（この理解が無いまま前回判断を誤ったため明記）:

- `docs/` は `src/` の**完全な鏡像**。両者の唯一の差分は `docs/.nojekyll` だけ。
- `src/` に無いファイルは `docs/` にも生成されない。「◯◯-bundle.js」のような結合物は**存在しない**。
- `src/` の1ファイルを直すと、`docs/` では同名の1ファイルだけが入れ替わる（`git diff --numstat` で src と docs が対になって1行ずつ変わるのが正常）。
- ゆえに**編集は `src/` のみ。`docs/` は生成物として扱い、手で触らない。** 公開は GitHub Pages が `main` の `/docs` をそのまま配信するだけ。

外部のビルド用依存は増やさず、Node.js の標準機能だけで生成します。

### 3.3 なぜ単元本文を「構造化 JavaScript」にするか

現時点の短い単元ノートは、リンク ID・図解設定・数式・表示ブロックを一緒に検査できる構造化 JavaScript を正本にします。これはブラウザが読み込む小さなデータモジュールであり、編集時・実行時ともに負荷の問題にはなりません。むしろ Markdown へ一律に移すと、段階式・文章題・図の説明を再び文字列として曖昧に扱う危険があります。

一方、300字を超える読み物や出典付きの長文は Markdown を採用できるようにします。ただし移行の条件は「Markdown にした」ことではなく、見出し・段落・数式・出典・図の参照をビルド時に検査できることです。短い単元の構造化データと、長文の Markdown を使い分けるハイブリッドを基本方針にします。

## 4. データの分け方

### 4.1 単元 (本文＝`content/lessons/<id>.js`、目次＝`lessons.js`)

単元は教材本文の最小単位です。**本文は1単元1ファイル**（`content/lessons/<id>.js`）に置き、範囲・学習段階・短い説明・要点・例・確認を持たせます。図解や問題との**関連づけ・表示順・分野（strand）・補足カードは本文には書かず**、目次の `lessons.js` が合成します（下記）。

```js
// src/content/lessons/linear-equations.js —— これが編集する正本
export const unit = {
  id: "linear-equations",
  stage: "方程式 1",          // 単元内の順番ラベル
  range: ["中1"],             // 学校段階（中1 / 中3 / 数I / 数A）
  title: "一次方程式を解く",
  summary: "左右から同じブロックを取り…",
  points: ["…", "…"],         // 要点は3つ以内が基本
  example: {                   // 例は §4.1.1 の5系統から選ぶ
    type: "walkthrough",
    intro: "…",
    steps: [
      { equation: "3x-5=16", note: "…" },
      { equation: "x=7", note: "…" },
    ],
    conclusion: "…",
  },
  check: "…",                  // 「小さく確かめる」一問
  // 任意の本文キー: checkExample?（別解の例） / model?（図。type は §4.2）
};
```

表示順・関連づけ・補足は目次の `lessons.js` が合成します。全単元を import し、表示順 `learningPath` と `lessonMetadata`（`strand`・`practiceIds`）を合わせて、各単元へ `order`（`learningPath` 順の連番）・`nextLessonId`・`labIds`（図解からの逆参照）・`strand`・`practiceIds`・`recommended{Lab,Practice,Next}Id`（＝「次の一手」）を付与します。補足カード（`context`）は同じ `lessons.js` の `lessonContexts` に単元idで登録します（本文ファイルではありません）。**だから単元本文ファイルには `order`・`strand`・`labIds`・`nextLessonId`・`context` を書きません。**

`range` は学校段階（`中1`、`中3`、`数I`、`数A`）だけに使い、`stage` は単元内の順番だけに使います。たとえば「数I」が二つ並んで別の意味に見える、といったラベルの混乱を避けます。

### 4.1.1 本文と例の表示ブロック

単元本文を単なる HTML 断片や「数式を含む一つの文字列」にはしません。見た目を決める前に、教材上の役割を次の型で明示します。

| 型 | 使う場面 | 表示の約束 |
| --- | --- | --- |
| 文字列（ふつうの説明） | 定義・要点・短い一言 | `=`・`\Rightarrow` を含めば自動で式に整形、無ければ本文表示。文中の変数はインライン数式 |
| `walkthrough` | 手順を1ステップずつ（最も多用） | 番号つきで各ステップに式（`equation`）か文（`text`）＋赤note（`note`）。`intro`・`conclusion` は任意 |
| `aligned-steps` | 計算過程・互除法 | `rows` を等号ぞろえで縦に並べる。別の式を一つの連鎖にしない |
| `word-problem` | 文章題から式を作る例 | `prompt`（問題文）→`explanation`（量の意味）→`equation`（式）を分けて示す |
| `narrative` | 図形の説明・短い比較 | `body` を先に読み、必要なら `equation` だけを独立表示する |

たとえば複数の等式をカンマでつないだ文章を自動で縦計算に変換してはいけません。互除法なら各行が別の等式、分配法則の途中式なら同じ式の変形、文章題なら式の前に状況説明が必要です。この区別をデータで持ち、表示側の推測に任せないことを品質基準にします。

### 4.2 図解 (`labs.js`)

図解は「遊べる部品」ではなく、特定の誤解をほどくための教材です。`id`、対応単元、初めに見せる具体物、確かめる不変量をデータとして持ちます。

```js
{
  id: "equation-blocks",
  title: "x袋と1ブロック",
  short: "方程式ブロック",          // 一覧・検索での短い見出し
  category: "方程式",
  lessonIds: ["linear-equations"],  // 必須・逆参照（この図解を使う単元）
  practiceIds: ["equation"],        // 必須・逆参照（対応する練習）
  objectIntro: "同じ数の1ブロックを、左右から同時に取る",
  observe: "左右の合計はいつも同じ",
  starterExample: "例：左に x袋1つと1ブロック2つ、右に1ブロック5つ。両側から1ブロック2つを取ると x=3 が見える。", // 必須
}
```

`lessonIds`・`practiceIds`・`starterExample` は必須（`check` が逆参照と存在を検査）。`hostId` フィールドは無く、キャンバス描画は次項のとおり `id` 対応で `labs-view.js` に書きます。

図解の表示と実際の操作コードは分けます。上のデータは「どの単元から何の図解へつなぐか」を担当し、キャンバス描画やスライダー処理は、現在は `assets/js/labs-view.js` 等の役割別モジュールにあります（`app.js` は初期化のみ）。この分割後も、教材データの参照方法は変わりません。

#### 4.2.1 応用図解の安全な扱い

ローン・会計・速さ・データのように日常や社会に接続する図解は、具体的な判断を勧めるページにはしません。画面で扱うのは、次のような一つの数値関係だけです。

| 題材 | 確かめる関係 | 画面に明記する範囲 |
| --- | --- | --- |
| ローン | 元金 × 年率 × 年数（単利の教材モデル） | 実際の契約・返済方式・商品比較には使わない |
| 会計 | 売上 − 変動費 − 固定費 = 利益 | 税金・在庫・事業判断は扱わない |
| 速さ | 道のり = 速さ × 時間 | 速さが一定の場面に限る |
| データ | 同じ平均でも散らばりは変わる | まず最小・最大の差を見て、分散・標準偏差は別単元へつなぐ |

各図解は、スライダーの現在値、図、説明文を同時に更新します。数だけを動かして「何を変えたのか」が分からなくならないことを、操作の必須条件にします。

### 4.3 問題 (`practice.js`)

問題は「出題器の設定」と「採点・解説の処理」を分けます。設定側には、初期難度・数の上限・前提単元・関連図解を置きます。

```js
{
  id: "simplify-radical",
  label: "平方数を外へ出す",          // 一覧の見出し（title ではない）
  lessonIds: ["simplify-roots"],      // 逆参照
  labIds: ["radical-simplifier"],     // 逆参照
  level: "はじめの一問",
  numberPolicy: "根号の中は100未満に収める", // 説明テキスト（コード値ではない）
}
```

`generator` フィールドは持ちません。モード `id` → 生成器の対応は `assets/js/practice-generators.js` の対応表で結びます。生成器は純関数で `{ modeLabel, title, prompt, steps: [{ label, question, hint, check, answer, choices?, example?, accept? }] }` を返し、`test-practice.mjs` が全モードの自己受理を検査します（`check` の一部）。

最初の問題は、原則として答えや途中計算が 2 桁程度に収まる設定にします。問題を解けない理由が、規則の理解ではなく繰り上がり・九九・大きな数の分解にならないようにするためです。慣れてから選べる「少し進んだ問題」を別に追加します。

### 4.4 学習マップ (`topics.js`)

学習マップは新しい本文を持たず、単元への入口です。`topics.js` は各 `unit` から**自動生成**するので、手で項目を足しません（新単元は `strand` と `range` を正しく設定すれば自動でマップに載ります）。

```js
// topics.js —— units から自動生成する（手動追加しない）
export const topics = units.map((unit) => ({
  id: `topic-${unit.id}`,
  lessonId: unit.id,
  category: categoryForLesson(unit),  // strand → キー（number/algebra/function/geometry/data/logic/sequence）
  level: levelForLesson(unit),        // range → 1〜5
  title: unit.title,
  description: unit.summary,
}));
```

`category` は日本語ラベルではなく**キー**で、`categoryForLesson(strand)` が決めます（表示名は `categoryLabels`）。`level` は `levelForLesson(range)` が決めます。**新しい `strand` 値を足したら `categoryForLesson`、新しい `range` 値を足したら `levelForLesson` を更新**しないと既定分類（algebra／level 1）に落ちます。これにより、単元名や範囲の修正は一か所で済みます。

### 4.5 読み物 (`stories.js`)

読み物は、単元本文を長くせずに「なぜその約束が必要か」を補う教材です。`type`、対応する単元・図解・問題、本文、出典、事実確認の状態を持たせます。

```js
{
  id: "equal-sign-keeps-balance",
  type: "rule",                        // rule / history / society
  menuTitle: "等号の役割",             // メニューの短い見出し
  title: "等号は「左右が同じ」の宣言",   // 必須
  lead: "等号は答えを書く記号ではなく、左右が等しいという約束です。", // 必須
  lessonIds: ["identities-equations", "linear-equations"],
  labIds: ["equation-lab"],
  practiceIds: ["equation"],
  sections: [                          // 必須。各節は { heading, body }
    { heading: "困る場面", body: "…" },
    { heading: "共有する約束", body: "…" },
  ],
  sources: [],                         // history / society は HTTPS 出典を1つ以上（必須）
  factCheck: { status: "checked", note: "外部の事実を含まない" }, // status は "checked" 必須
}
```

肖像を使う読み物は `portraits: [{ src, alt, caption }]` を付け、画像ファイルは実在必須（`check` が確認）。

人物史・社会・統計を扱う `history` / `society` の読み物では、`sources` に HTTPS の出典を少なくとも一つ入れます。出典と `factCheck.status` はビルド前の内容検査で確認します。

## 5. 相互リンクのルール

リンクはすべて固定 ID を介して作り、表示文言や配列の順番へ依存させません。

| 出発点 | 行き先 | 表示する導線 |
| --- | --- | --- |
| 範囲 | 単元 | 「この範囲の単元を見る」 |
| 単元 | 図解 | 「x袋と1ブロックを動かす」 |
| 単元 | 問題 | 「この単元の一問を解く」 |
| 図解 | 単元 | 「説明を読む」 |
| 図解 | 問題 | 「この図解のルールで解く」 |
| 問題 | 図解 | 「図解に戻る」 |
| 学習マップ | 単元 | 「単元を開く」 |

一つの単元に関連する図解・問題が複数ある場合でも、最初に表示するのは「おすすめ 1 つ」です。残りは「ほかの図解」「練習を増やす」にしまい、スマートフォンで選択肢が増えすぎないようにします。

## 6. 画面構成とスマートフォンでの優先順位

ハッシュで開くページを、次の九つに整理します（`assets/js/nav.js` の `pageIds` が正）。

| URL | 役割 | 画面上で最初に見せる内容 |
| --- | --- | --- |
| `#home` | はじめに | 今日の入口を一つだけ |
| `#scope` | 範囲 | 今の単元が中学・数I・数Aのどこか |
| `#lessons/<lesson-id>` | 単元 | 見出し、3つ以内の要点、例、次の一手 |
| `#labs/<lab-id>` | 図解 | 具体物、操作、観察すること |
| `#practice/<practice-id>` | 問題 | 一問、今の手順、ヒント |
| `#stories/<story-id>` | 読み物 | ルールの役割、短い確認、教材へ戻る導線 |
| `#figures/<figure-id>` | 数学者図鑑 | 人物、業績、関連する単元・読み物 |
| `#map` | 学習マップ | 次に選べる少数の単元 |
| `#search` | 検索 | 単元・図解・読み物・数学者を横断して探す |

現行の `#lessons` のような大分類ハッシュも残し、旧リンクを壊しません。詳細ルートは単元 `#lessons/linear-equations`、範囲で絞る `#lessons/range/<range>`、図鑑 `#figures/<figure-id>` のようにします。

### モバイルの原則

1. 一画面に一つの主目的だけを置く。
2. 単元本文は「要点 3 個・例 1 個・次の一手 1 個」を基本とする。
3. 長い単元一覧は、常時表示せず「単元を選ぶ」ボタンから開く。
4. ページ送りは現在地と前後一つずつを優先し、全ページ番号は補助にする。
5. 図解は説明を読んでから操作できる順序にし、キャンバスだけを先に置かない。

## 7. 数式表記を壊さないための規約

数式は MathJax の TeX として保存します。表示用 HTML を組み立てるときに `<` や `>` が HTML タグとして扱われないよう、本文・例・要点・ラベルは必ず `escapeHtml()` を通します。

さらに、ビルド時に `scripts/validate-content.mjs`（＋練習は `scripts/test-practice.mjs`）が内容を検査します（`npm run check` と `npm run build` の両方が通す）。**検査項目の正本はスクリプト本体**で、主なものは次のとおり:

- 単元・図解・問題・学習マップ・読み物・数学者・用語・検索同義語の ID／語が重複していないこと
- 相互リンク（`lessonId`・`labId`・`practiceId`・`storyId`・図鑑の関連参照、`context.connections` の逆参照）がすべて実在すること
- すべての単元に「次の一手」（`nextLessonId`／`labIds`／`practiceIds`、および `recommended*`）が最低一つあること
- `example` が型（文字列／`aligned-steps`／`walkthrough`／`word-problem`／`narrative`）ごとの構造を満たすこと（式にカンマを入れない等）
- `context` を持つなら `why`・`definitions`・`connections` の3つがそろい、逆参照が通ること
- 読み物が出典（history/society は HTTPS 必須）と事実確認（`factCheck.status: "checked"`）を備え、肖像ファイルが実在すること
- 数式を含む文字列が `\(...\)`／`\[...\]` で区切られ、教材データに生の HTML タグが混ざっていないこと
- `docs/`（および `src/`）の `index.html` が参照するローカル CSS・JS・画像がすべて存在すること
- 練習問題が各モードで自己受理できること（`test-practice.mjs`）

## 8. 移行の順番

1. この設計文書を正本にし、出力先を `docs/` に決める。
2. 現在の静的ファイルを `src/` へ移し、同じ内容を `docs/` へ生成できるようにする。
3. `units`、`topics`、図解カタログ、問題モード設定を `src/content/` へ切り出す。
4. `app.js` を、ルーティング・表示・図解・問題生成の役割ごとに分ける。
5. 相互リンクを ID ベースへ統一し、旧ハッシュへの互換を残す。
6. ビルドと内容検査を通してから、GitHub Pages の対象を `main /docs` に変更する。

この移行中は `docs/` が公開物ですが、GitHub にはプッシュしません。公開前に、リンク切れ・数式・スマートフォンでの単元導線をまとめて確認します。

## 9. 完了の目安

- `src/` だけを編集すれば、`docs/` の完成物を再現できる。
- 単元・図解・問題のどこからでも、関連する次の教材へ移れる。
- 初心者向けの問題設定がデータとして明示されている。
- 数Iと数Aの範囲ラベルと、単元内の順序ラベルが混ざらない。
- GitHub Pages は `main` ブランチの `/docs` を配信するだけで動く。

## 10. 表記と文字組みのルール

画面上の役割で書体を分けます。目次・ボタン・範囲ラベル・見出しは、選びやすさを優先してゴシック系にします。説明文・要点・例・ヒントは、PCでは18pxを基準にした明朝系とし、落ち着いて読める行間を確保します。

変数・演算・関数名は、文中であってもMathJaxの数式書体にそろえます。たとえば `x袋` は「数式の `x` + 日本語の袋」として表示します。教材データにすでに含まれるTeX表記はそのまま保持し、プレーンテキストの孤立した `x`、`y`、`a`、`b`、`n`、`sin`、`cos`、`tan` は表示時に数式化します。

ただしHTMLの`select`の選択肢内はMathJaxが組版しないため、数式を厳密に見せる必要がある操作は、将来ボタン型の選択UIなどへ移します。

## 11. ルールの理由・背景・社会との接続

### 11.1 前提

この教材では、公式や定義を「すでにある決まり」としてだけ提示しません。学習者が「なぜそう決めるのか」で止まることを、理解不足ではなく自然な問いとして扱います。

各ルールは、可能な範囲で次の順に短く示します。

```text
困る場面・そろえたいこと
    ↓
共有する約束・定義
    ↓
その約束で解けること
    ↓
手を動かす小さな確認
```

たとえば計算順序は、「同じ式を見た人が同じ答えにたどり着けるようにする共通の読み方」として置きます。かっこは、その共通の読み方の一部を意図的に変更するときの印です。ここで扱うのは厳密な数学史の断定ではなく、学習者がルールの役割をつかむための説明です。歴史的事実を述べる場合は、出典を確認し、後づけの単純な物語にしません。

### 11.2 単元に追加する文脈データ

単元の本文とは別に、`lessons.js` の `lessonContexts` に**単元 id をキーにして**次の短い要素を登録します（本文ファイルには書かない）。すべてを毎回開かず、必要な子が「なぜ？」を開ける構成にします。

```js
// src/content/lessons.js の lessonContexts に、単元 id をキーにして登録する
const lessonContexts = {
  "integer-rules": {
    why: {
      question: "どうして計算する順番を決めるの？",
      answer: "式を読む人によって答えが変わらないよう、共通の読み方を決めているから。",
      tryIt: "2+3×4 を、先に足す場合と先にかける場合で比べる",
    },
    definitions: [
      {
        term: "式",
        meaning: "数や記号を、計算の約束に従って並べたもの",
        example: "3+4×2 は、×を先にして 3+8=11",
        boundary: "等号で『成り立つか』を問うと方程式になる",
      },
    ],
    connections: [
      {
        kind: "daily-life",          // 分類ラベル（history/model/notation… 現状は描画に未使用）
        title: "会計の合計",
        summary: "計算順序をそろえると、複数の人が同じ集計結果を確かめられる",
        // labId / storyId / practiceId を付けるとカードにボタンが出る（逆参照検査あり）
      },
    ],
    storyIds: ["shared-notation"],   // 「どこで役立つ？」に並べる読み物ボタン
  },
  // ...他の単元
};
```

`why` は「ルールの役割」を、`definitions` は「言葉をどう使うか・どこまでを指すか」を、`connections` は「何を扱えるようになるか」を担当します。正解を出すために全てを読ませるのではなく、立ち止まったときの補助線にします。

### 11.3 画面の置き方

単元ページでは、要点の直後に常時長文を置きません。次の二つを短い開閉式のカードとして置きます。

| カード | 入口の文 | 中身 | 量 |
| --- | --- | --- | --- |
| なぜこの約束？ | 「もし決めなかったら、何が困る？」 | 困る場面 → 約束 → 小さな比較 | 2〜4文 + 1操作 |
| どこで役立つ？ | 「この考え方で、何が見える？」 | 身近な場面・科学・仕事への接続 | 2〜4文 + 図解リンク |

用語は別の小カード「言葉をほどく」にまとめ、定義だけで終わらせず、使う例と使わない例を一つずつ示します。読み物を開かない学習者でも単元と問題は進められ、読み物を開く学習者には理解の出口を増やす設計です。

### 11.4 読み物ページ

単元とは別に `#stories/<story-id>` の読み物ページを設けます。読み物は単元順の長い連続講義にせず、次の三種類を混ぜます。

| 種類 | ねらい | 例 |
| --- | --- | --- |
| ルールの物語 | 約束が必要になる場面を知る | 計算順序、負の数、等号、座標 |
| 人と発見の物語 | 数学者・科学者がどんな問いを扱ったか知る | デカルトと座標、ガウスと数、フローレンス・ナイチンゲールと統計 |
| 社会と仕事の物語 | 数学が現代の判断・説明にどう使われるか知る | ローンの返済、利益と損益分岐、データのばらつき、測定と誤差 |

人物史は「一人の天才が突然作った」とは見せず、当時の問題・先行する考え・今の使われ方を一緒に扱います。金融や医療などの題材は、個別の意思決定を勧めるものではなく、数値の関係を読む練習として扱います。

### 11.5 動かせる応用例

難しい現実の話題も、最初は数値を操作するだけで関係が見える図解にします。

- ローン: 元金・年率・回数のバーを動かし、月々の返済額と総返済額の違いを見る
- 会計: 売価・原価・固定費・販売数のバーを動かし、利益が0になる点を読む
- 科学: 距離・時間・速さを動かし、表・グラフ・式が同じ関係を示すことを見る
- データ: 外れ値を一つ動かし、平均・中央値・散らばりがどう変わるかを見る

各図解は、数式を先に覚えさせるためではなく、「何を変えると何が動くか」を観察するために置きます。数値は初めは小さく、単位・前提・現実との違いも短く明示します。

### 11.6 追加時の品質基準

1. 「なぜ？」への答えは、権威ではなく困る場面・共通の目的・使えるようになることから始める。
2. 定義には、使う例と境目の例を添える。
3. 歴史・人物・社会の事実は、信頼できる一次資料・公的機関・博物館・大学などで確認し、読み物に出典を示す。
4. 一枚に情報を詰め込まず、単元本文・理由カード・応用図解・読み物を分ける。
5. 読み物を読まないことが問題演習の障害にならないようにする。
